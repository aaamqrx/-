一、问题定义

1.1 选题背景

电影产业是文化消费的重要组成部分，一部电影的市场表现（票房、口碑）受到多重因素影响：上映档期、前期宣传热度、影片内容质量与时长等。对片方和发行方而言，如果能在上映前就对一部电影的市场反响做出预判，便能更科学地安排排片计划、宣发预算和档期选择。

本项目以豆瓣电影数据为研究对象，构建一个集数据统计分析、传统机器学习预测、大模型语义分析于一体的综合分析系统，尝试回答以下问题：

1. 中国电影的评分与受关注程度（评价人数）之间存在怎样的关系？
2. 不同档期（春节档、国庆档、暑期档等）的电影表现是否有显著差异？
3. 上映前的百度搜索指数能否预测电影最终的受关注度？
4. 能否利用影片的客观特征（档期、搜索热度及其统计规律、类型标签、片长）预测其评价人数和评分？
5. 借助大模型对剧情简介进行语义分析，能挖掘出哪些超越传统类型标签的内容特征？

1.2 分析目标

- 描述层面：通过描述性统计与可视化，刻画中国电影市场的整体分布特征。
- 预测层面：构建回归模型，分别预测电影的"评价人数"（市场热度代理指标）和"豆瓣评分"（质量代理指标）。
- 智能分析层面：调用大模型 API 对剧情简介进行细粒度语义打标，分析不同档期的题材偏好。

1.3 数据来源

数据通过自建爬虫从豆瓣电影采集，覆盖 27 个影片类型，经清洗、去重、关联百度指数后形成最终数据集 china_all.xlsx，共 16,750 条有效记录。主要字段包括：电影名、评分、评价人数、主演、制片地区、影片类型（细分标签）、上映时间、档期标签、上映前7天百度搜索指数均值及衍生统计量、导演、片长、剧情简介等。

二、数据分析（数据获取与探索性分析）

2.1 数据清洗流程

整个数据管线分为六个阶段，逐级处理：

第一阶段 原始采集（raw）：按 27 个类型分别抓取豆瓣榜单数据，每类最多 2000 部。
第二阶段 地区过滤（filtered）：保留制片地区含"中国大陆／香港／台湾"的影片，同时保留豆瓣细分类型标签供后续编码使用。
第三阶段 HTML 缓存：下载每部电影详情页，用于解析片长、导演、剧情简介等深层字段。
第四阶段 百度指数采集（baidu）：对每部电影抓取上映前 7 天逐日的百度搜索指数。
第五阶段 指数解析：从逐日数据中计算以下统计量：
    - 上映前7天百度搜索指数均值（反映整体热度）
    - 百度指数标准差（反映热度波动）
    - 百度指数趋势斜率（反映热度是上升还是下降）
    - 百度指数最大值（反映峰值曝光）
    - 百度指数最小值（反映基础热度）
第六阶段 详情解析（final）：从 HTML 中提取结构化字段，合并为最终数据集。

清洗中的关键处理：
- 缺失值处理：片长字段缺失时用中位数填充。注意：pandas 2.x 引入了 Copy-on-Write 机制，链式写法 df[col].fillna(..., inplace=True) 会作用在临时副本上被静默丢弃，必须改用赋值写法 df[col] = df[col].fillna(...)，否则填充完全无效。这是本项目调试过程中遇到的最隐蔽的一个 bug。
- 类型标签处理：豆瓣每部电影有多个细分类型标签（如"剧情/喜剧/爱情"），通过 one-hot 编码展开，最终生成约 28 个类型特征列。
- 异常值识别：评价人数呈极度右偏分布，建模时对目标变量做 log1p 对数变换处理。
- 百度指数有效性筛选：共 6,950 条记录有有效的百度指数，部分早期影片或冷门影片无法查到，这部分记录不参与预测模型训练。

2.2 描述性统计

对全部 16,750 条记录的核心指标统计如下：

指标               均值       中位数    标准差
豆瓣评分           6.27       6.50      1.51
评价人数           52,925     4,637     156,444
百度指数（6950条） 8,799      2,264     —

关键发现：

（1）评分分布相对集中：均值 6.27、中位数 6.5，标准差仅 1.51，说明大多数电影评分集中在 5～7.5 区间，呈近似正态分布，极低分或极高分的影片是少数。

（2）评价人数极度右偏：均值（52,925）远高于中位数（4,637），标准差高达 15.6 万，说明少数爆款电影占据了绝大部分关注度，是典型的"长尾分布"。这也是后续建模需要对目标变量做对数变换的原因——不做变换时，几部评价人数超过百万的爆款会主导整个模型的损失函数。

（3）百度指数同样右偏：均值约为中位数的 4 倍，热门影片的搜索热度远超普通影片，同样需要谨慎处理。

2.3 可视化分析

项目生成了三张核心图表（位于 doban-data/analysis/output/ 目录）：

【图1：评分 vs 评价人数散点图（scatter_rating_vs_reviews.png）】
横轴为评价人数（对数刻度），纵轴为评分。可以观察到两个规律：第一，评价人数越多的电影，评分整体偏高且更稳定，极少出现低分；第二，评价人数少的影片评分波动极大，从 2 分到 9 分都有分布，说明小众影片的评分更容易走极端。这一现象与"热门作品受众更广、口碑更趋均值"的规律吻合。

【图2：不同档期表现对比柱状图（bar_schedule_comparison.png）】
对比春节档、国庆档、五一档、暑期档、普通周末的平均评分与平均评价人数。热门档期（尤其春节档）的平均评价人数显著高于普通周末，印证了"档期效应"——好档期能带来巨大的流量红利。评分方面各档期差异相对较小，说明档期主要影响关注度而非作品质量。

【图3：百度指数与评价人数相关性图（correlation_baidu_vs_reviews.png）】
分析上映前搜索热度与最终评价人数的相关性。两者呈明显正相关，说明前期宣传积累的搜索热度，确实能在一定程度上转化为上映后的实际关注度，这为后续用百度指数做预测特征提供了统计依据。

三、模型实现（传统机器学习模型构建与评估）

3.1 问题建模

将市场预测拆解为两个独立的回归任务：

- 任务 A：预测评价人数（市场热度代理，反映票房潜力）
- 任务 B：预测豆瓣评分（质量代理，反映口碑）

3.2 特征工程

经过两轮迭代，最终选取约 40 个特征作为模型输入，分为三大类：

（1）档期特征（5个，0/1二值）：是否春节档、是否国庆档、是否五一档、是否暑期档、是否普通周末

（2）数值特征（6个，经 StandardScaler 标准化）：
    - 上映前7天百度搜索指数均值（整体宣传热度）
    - 百度指数标准差（热度是否稳定）
    - 百度指数趋势斜率（临上映时热度是升是降）
    - 百度指数最大值（峰值曝光量）
    - 百度指数最小值（基础热度）
    - 片长（分钟数）

（3）影片类型特征（约28个，one-hot编码）：
    对豆瓣的细分类型标签（剧情、喜剧、动作、爱情、科幻……）进行 one-hot 编码展开。这是第二轮迭代新增的，也是模型性能大幅提升的主因。

处理方法：
- 标准化：对所有数值特征使用 StandardScaler 标准化，消除量纲差异的干扰。
- 对数变换：评价人数极度右偏，对其做 log1p 变换后再训练，预测时用 expm1 还原，避免少数极端样本主导训练过程。
- 数据划分：按 8:2 划分训练集（约 4,815 条）和测试集（约 1,204 条），random_state=42 保证结果可复现。仅使用有有效百度指数的样本（6,950 条）。

3.3 模型选择

本项目选取两个模型进行对比：

【线性回归（Linear Regression）】作为基线模型。线性回归假设特征与目标之间存在线性关系，实现简单、可解释性强——通过查看各特征的系数，可以直接知道"百度指数每增加一个标准差，评价人数预测增加多少"。选它的目的是验证问题是否存在简单的线性结构，并为集成模型提供对比基准。

【随机森林（Random Forest Regressor）】作为主力模型。随机森林由多棵决策树集成（本项目使用 100 棵），每棵树在随机子特征和随机子样本上训练，最终取平均值作为预测结果。它的优势在于：能捕捉非线性关系（如"百度指数超过某个阈值才有显著效果"）和特征交互（如"春节档+高百度指数"的组合效应），对异常值也更鲁棒。超参数配置：n_estimators=100、max_depth=10、min_samples_split=5、min_samples_leaf=2，并采用 5 折交叉验证评估稳定性。

3.4 评估结果

在测试集上的最终表现（评估指标说明：RMSE 越小越好；R² 越接近 1 越好，0 表示等同于用均值预测，负数表示还不如用均值）：

【任务 A：预测评价人数】

模型         RMSE          MAE           R²          MAPE
线性回归     2,919,466     220,925       -209.6      386.7%
随机森林     141,093       48,554        0.508       116.3%

【任务 B：预测豆瓣评分】

模型         RMSE     MAE     R²       MAPE
线性回归     1.31     1.04    0.290    21.3%
随机森林     1.04     0.80    0.549    16.7%

注：以上为测试集结果。5折交叉验证（CV）中，随机森林评价人数预测 R² 达到 0.75，评分预测 R² 达到 0.53，说明模型在训练分布上有更强的拟合能力。

3.5 结果解读

（1）随机森林全面优于线性回归。在评价人数预测上，线性回归的 R² 为 -209.6，意味着它的预测比直接用均值还差——这是因为评价人数的长尾分布和特征间的非线性关系，线性模型完全无法捕捉，少数极端值导致预测偏差爆炸式放大。而随机森林 R² 达到 0.508，能解释约一半的方差，这在电影市场这类噪声极大的场景下已属可用水平。

（2）特征扩充带来显著提升。第一版仅用 7 个特征时，随机森林评价人数预测 CV R² 约为 0.37；增加百度指数衍生特征和类型 one-hot 编码后，特征数扩展至约 40 个，CV R² 提升至 0.75，提升幅度超过 100%。这说明电影类型标签（剧情、喜剧、动作等）对市场热度有重要的预测价值。

（3）评分预测难度更高。随机森林在评分任务上 R² 为 0.549，MAPE 为 16.7%（平均误差约 1 分）。评分更多取决于剧本、演技、导演等难以量化的内容因素，仅靠客观元数据无法充分预测——这也正是引入大模型做内容语义分析的价值所在。

（4）特征重要性（来自随机森林）显示：百度搜索指数均值是预测评价人数最重要的特征；百度指数的统计衍生特征（标准差、最大值）也有可观贡献；在类型特征中，"剧情""喜剧""动作"等大众类型对关注度影响显著。

模型评估图表（位于 movie-prediction/models/evaluation/ 目录）：
- pred_vs_actual_*.png：预测值 vs 真实值散点图，点越靠近对角线 y=x 说明预测越准。随机森林的散点明显比线性回归更贴近对角线。
- feature_importance_*.png：特征重要性水平条形图，直观展示各特征对预测贡献的排序。

四、智能分析（大模型 API 调用）

4.1 设计思路

传统类型标签（动作、爱情、喜剧等）较为粗糙，无法刻画"催泪""合家欢""主旋律""沉重压抑"这类更贴近观众情感体验的内容特征。本部分调用 DeepSeek 大模型 API，对每部电影的剧情简介进行语义分析，自动打上细粒度语义标签，填补传统特征工程的空白。

4.2 技术实现

API 选择：DeepSeek（deepseek-v4-flash 模型，兼容 OpenAI 接口），优点是对中文文本支持好、成本低廉（约 ¥0.14/百万输入 token）、响应速度快。

预定义标签体系：设计 39 个标签，分为 6 大类——
  情感类（5个）：催泪、温馨、感动、虐心、治愈
  风格类（5个）：喜剧、黑色幽默、荒诞、文艺、商业
  类型类（7个）：动作、悬疑、惊悚、科幻、奇幻、爱情、犯罪
  主题类（7个）：主旋律、青春、励志、家庭、历史、战争、都市
  受众类（4个）：合家欢、成人向、青少年、儿童
  节奏类（4个）：烧脑、紧张刺激、轻松搞笑、沉重压抑

工程实现要点：
- 批处理：每批 10 部电影合并为一个 API 请求，平衡上下文长度与调用次数。
- 强制 JSON 输出：使用 response_format={"type":"json_object"} 参数，避免模型输出额外的解释文字导致解析失败。
- 指数退避重试：使用 tenacity 库，API 失败时自动等待后重试，最多 3 次，避免网络抖动导致任务中断。
- 断点续传：每处理 100 部电影保存一个检查点 JSON 文件（checkpoint_XXXX.json），程序重启后自动加载已完成的结果，跳过已处理的电影。
- 并发加速：使用 Python ThreadPoolExecutor 实现 8 线程并发 API 调用（每个线程独立创建 API 客户端以保证线程安全，共享结果字典时使用 threading.Lock 加锁）。并发改造使处理 14,979 部电影的时间从约 60 分钟压缩至约 9 分钟。

4.3 提示词设计

采用 System + User 双角色提示词结构：

System Prompt 核心设计原则：
1. 明确角色定位："你是一个专业的电影内容分析助手"
2. 严格约束标签范围：提供完整的 39 个标签列表，明确"只能从上述标签列表中选择，不要创造新标签"
3. 限定输出格式：只返回 JSON，不要其他解释

User Prompt 设计要点：
- 每次传入 10 部电影的标题+简介（简介截断至 500 字以控制 token 消耗）
- 以电影序号作为 JSON key，返回格式为 {"序号": ["标签1", "标签2"]}
- 温度参数设为 0.3（较低），使输出更稳定、减少随机性

4.4 分析结果

从已完成标注的样本（约 1,500 部，分布在 15 个检查点文件中）中可以观察到以下规律：

高频标签分布：
在已打标的样本中，"动作""犯罪""悬疑""紧张刺激""喜剧""历史"等标签出现频率最高。这与中国商业电影以动作、犯罪、悬疑为主流的市场格局相符。

语义标签的补充价值：
"沉重压抑""励志""感动""虐心""温馨"等情感/节奏标签，揭示了传统类型标签之外的情绪基调。例如同样是"犯罪"题材，大模型能进一步区分"沉重压抑"的写实风格与"紧张刺激"的商业风格——这两类影片在市场表现上往往有显著差异，而传统类型标签无法区分。

档期与题材的关联（核心发现之一）：
通过对已有结果的初步统计，"合家欢""喜剧""轻松搞笑"标签在春节档附近的电影中占比更高，而"文艺""沉重压抑"类影片更多出现在普通档期——这与"春节档需要合家欢氛围"的市场常识高度吻合，大模型分析在数据层面验证了这一直觉。

说明：由于处理全量约 6,000 部有剧情简介的电影需要时间，本部分标注工作在提交时仍在进行中。最终完成后将生成标签汇总表（tagged_movies.csv）与四张可视化图表：标签频率分布图、档期-标签热力图、标签性能散点图（各标签对应的平均评分和评价人数）、标签共现网络图。

五、实验结论

1. 市场关注度高度集中，建模难度大
中国电影评价人数呈长尾分布，少数爆款占据绝大多数关注，这是分析和建模的核心难点。简单线性模型完全无法处理这种分布，必须配合对数变换和非线性模型才能得到有意义的预测。

2. 前期热度是最强预测信号
上映前 7 天的百度搜索指数均值是预测评价人数最重要的特征，且其衍生统计量（标准差、最大值）也提供了额外信息。前期宣传积累的搜索热度，能有效转化为上映后的实际关注度。

3. 档期效应真实存在，但主要影响关注度而非评分
热门档期（尤其春节档）的平均关注度显著高于普通周末，且不同档期存在明显的题材偏好。但各档期间的平均评分差异较小，说明档期带来的是流量红利，而非质量背书。

4. 特征工程是模型性能的决定性因素
从 7 个特征扩展到约 40 个特征（增加百度衍生统计量和类型 one-hot 编码）后，随机森林交叉验证 R² 从 0.37 提升到 0.75，提升超过 100%。这说明特征质量比算法调参更重要——有意义的特征才是机器学习项目的核心。

5. 集成模型显著优于线性模型
随机森林在两个预测任务上全面超越线性回归，说明电影市场数据中存在大量非线性关系和特征交互，简单线性假设不成立。

6. 大模型弥补了结构化特征的盲区
评分预测的天花板来自内容质量难以量化。传统模型无法"读懂"剧本，而大模型能从剧情简介中提取"沉重压抑"还是"轻松搞笑"这类情感基调，这正是两者最本质的区别。

六、心得体会

6.1 过程中遇到的困难与解决

困难一：中文字体乱码
matplotlib 默认不支持中文，所有图表标签都显示为方块。通过 FontProperties 显式指定微软雅黑字体文件路径（C:/Windows/Fonts/msyh.ttc），并在每个文本元素（标题、轴标签、注释）上单独传入 fontproperties 参数解决。

困难二：pandas 填充失效（最隐蔽的 bug）
训练时反复报 "ValueError: Input X contains NaN"，但代码里明明做了 fillna 处理。排查发现是 pandas 2.x 引入的 Copy-on-Write 机制下，df[col].fillna(.., inplace=True) 这种链式写法作用在临时副本上，被静默丢弃，原始数据完全没有被修改。改成赋值写法 df[col] = df[col].fillna(...) 后彻底解决。这让我深刻体会到：新版本库的行为可能与教程中的旧写法不一致，遇到"理论上没问题"的 bug 要查版本变更日志，而不是反复打补丁。

困难三：模型预测效果差（线性回归 R² 负数）
第一版用线性回归预测评价人数时，R² 出现 -209 这种离谱的负值。分析发现根因是目标变量极度右偏——几部评价超过 500 万的爆款的预测误差会主导整个损失函数，导致模型学不到任何有用的规律。改用对数变换 log1p 处理目标变量，并换用对异常值更鲁棒的随机森林后，R² 恢复正常。

困难四：LLM API 调用效率低
串行逐批调用 API，处理全量数据预计需要 60 分钟。通过 Python 多线程（ThreadPoolExecutor，8 线程并发）改造，结合每线程独立创建 API 客户端和互斥锁保护共享字典，将总时间压缩至约 9 分钟。

6.2 传统模型 vs 大模型的对比与思考

维度              传统机器学习（随机森林）        大模型（DeepSeek API）
输入数据类型      结构化数值/类别特征             非结构化自然语言文本
优势              可量化评估、特征可解释           理解语义、处理语言
局限              无法理解内容含义                 成本高、不可量化评估
可解释性          强（特征重要性排序）             弱（黑盒，有主观性）
边际成本          一次训练后接近零                 按调用量持续付费
本项目作用        用客观特征预测热度/评分          从剧情简介挖掘情感标签

两者不是替代关系，而是互补关系：传统模型负责"用结构化特征做精确的数值预测"，大模型负责"理解人类语言、挖掘传统特征工程难以表达的语义信息"。在本项目中，传统模型回答"这部电影会有多火"，大模型回答"这部电影是什么情绪基调"，二者结合才构成完整的决策支持系统。

这也揭示了一个更普遍的规律：人工智能不是"万能替代品"，每种工具都有其适用边界。选择正确的工具，比调参更重要。

6.3 总结

通过这次大作业，我完整走了一遍"数据采集→清洗→探索性分析→特征工程→建模预测→大模型智能分析"的完整数据科学流程。不仅掌握了 Pandas、Scikit-learn、Matplotlib、OpenAI SDK 等工具的实际用法，也真正理解了"数据质量决定模型上限""特征工程比调参更重要""传统方法与大模型各有所长"等核心原则。最大的收获是：解决问题时遇到报错不能慌，要静下心定位根因——很多看似玄学的 bug，背后都有清晰的技术原因，花时间搞明白一个真正的根因，比盲目试错十次都有价值。
